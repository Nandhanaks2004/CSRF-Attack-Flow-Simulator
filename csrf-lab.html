How Authentication Works
Sessions, cookies, and how servers identify you
When you log in to a website, the server creates a session and gives your browser a session cookie. Every future request includes this cookie — that's how the server knows who you are.

KEY POINT

The cookie is the server's way of saying "I know who you are." As long as the cookie is valid, the server trusts every request that carries it.

THE SILENT PROBLEM

The browser does this automatically and silently — you never see it happen. This is a feature for convenience, but it becomes a vulnerability when requests can be triggered by a third party.

✓ INTENDED
Alice clicks "Transfer Money" on bank.com → browser attaches cookie → server trusts Alice → transfer done
✗ EXPLOITABLE
Attacker tricks Alice's browser into sending that same request → browser still attaches cookie → server still trusts → CSRF

The Core Problem
Identity vs. Intention — what servers actually check
The fundamental flaw: The server checks who you are (via cookie), but never checks if you intended to make that request.
WHAT THE SERVER SEES
REQUEST FROM ALICE (NORMAL)
HTTP REQUEST — NORMAL
POST /transfer HTTP/1.1
Host: bank.com
Cookie: session=a1b2c3
Referer: https://bank.com
amount=100&to=bob
✓ Cookie valid → Request accepted
FORGED REQUEST (CSRF ATTACK)
HTTP REQUEST — FORGED
POST /transfer HTTP/1.1
Host: bank.com
Cookie: session=a1b2c3 ← same!
Referer: https://evil.com ← different
amount=10000&to=attacker
✓ Cookie valid → Request ALSO accepted ← THE BUG
THE MISSING CHECK
✓ SERVER CHECKS
  • Is the cookie valid?
  • Is the session active?
  • Is the user authenticated?
✗ SERVER DOES NOT CHECK
  • Did the user intend this?
  • Did user click a button?
  • Is request from trusted origin?
THE CONSEQUENCE
An attacker who can trigger a browser request gets the same server trust as the legitimate user.

What is CSRF?
Cross-Site Request Forgery explained simply
CSRF (Cross-Site Request Forgery) happens when an attacker tricks your browser into sending a request to a site where you're already logged in — without your knowledge or intention.
THE THREE-WORD DEFINITION
🔀
CROSS-SITE
The attack comes from a different website (evil.com) targeting another (bank.com)
📤
REQUEST
An HTTP request (GET or POST) is made to the target site — not just JavaScript execution
🎭
FORGERY
The request is forged — it appears to come from the legitimate user, but was made without their intent
SIMPLE ANALOGY

Imagine someone forges your signature on a check while you're asleep. The bank sees a valid signature and processes it — they never asked if you signed it willingly.

🏦 Bank = Web Server
✍️ Signature = Session Cookie
😴 Asleep = You don't know it happened
🕵️ Forger = Attacker
WHAT CSRF IS NOT
  • Not about stealing passwords
  • Not JavaScript injection (that's XSS)
  • Not about reading data (attacker is blind)
  • Not about breaking encryption
  • About abusing browser trust

ATTACKER MINDSET
🔍
"Is the target user logged in?"
Is there a session that will be active when my victim visits my site? Banking, social media, email — users stay logged in for hours.
🍪
"Will the browser automatically send cookies?"
Yes — unless SameSite cookie protection is in place. Older sites without SameSite are vulnerable.
⚡
"Can I trigger a cross-site request?"
Forms, images, iframes, and links all trigger HTTP requests. I just need to get the victim to load my page.
🛡
"Is there CSRF protection?"
Does the form include a hidden token? Is the token validated server-side? Many older applications skip this.

CSRF vs XSS
Two different attacks, often confused
CSRF and XSS are often mentioned together, but they are fundamentally different attacks. Understanding the distinction is crucial.
SIDE-BY-SIDE COMPARISON
ASPECTCSRFXSS
What it does Makes unwanted requests on user's behalf Executes malicious scripts in browser
JavaScript needed? No — pure HTML form works Yes — requires script injection
Can read response? No — attacker is blind Yes — full DOM access
Exploits Browser's automatic cookie sending Server's failure to sanitize input
Origin Attack comes from different site Attack injected into target site
User must be logged in? Yes — needs valid session Not necessarily
Primary defense CSRF tokens, SameSite cookies Input sanitization, CSP
Can steal cookies? No — can't read responses Yes — document.cookie
CSRF IN ONE SENTENCE

"Your browser does something you didn't ask it to do, using credentials it already has."

XSS IN ONE SENTENCE

"Malicious code runs inside a trusted website, as if it were part of the site."

COMBINED ATTACK (THE WORST CASE)

When an attacker has XSS on a site, they can also bypass CSRF protection — because they can read the CSRF token from the DOM and include it in their forged requests.

XSS injected
→
Script reads CSRF token
→
Includes token in forged request
→
CSRF protection bypassed!

How to stop CSRF at every layer

HOW CSRF TOKENS WORK

When the server serves a form, it embeds a random secret token. When the form is submitted, the token must be present and valid. An attacker's page cannot know this token.

protected form HTML
<form action="/transfer" method="POST">
 <input name="amount" value="100">
 <input type="hidden"
  name="csrf_token"
  value="xK9mP2...">
</form>
TOKEN VALIDATION DEMO
SIMULATE REQUEST TYPE
SERVER LOGIC
// Server validates every sensitive request
if (req.body.csrf_token === session.csrf_token) {
  // proceed
} else { reject(403 Forbidden) }
SAMESITE COOKIE ATTRIBUTE

The SameSite attribute tells the browser when to attach cookies to cross-site requests. It's the simplest modern defense against CSRF.

SameSite=None
Cookie sent in ALL cross-site requests. This is the old default — CSRF is possible.
VULNERABLE
SameSite=Lax
Cookie sent for top-level navigations (link clicks) but not for images, iframes, forms. The modern default.
PARTIAL PROTECTION
SameSite=Strict
Cookie NEVER sent in cross-site requests. The strongest protection — but may break some OAuth flows.
BEST PROTECTION
secure cookie configuration
Set-Cookie: session=abc123; SameSite=Strict; HttpOnly; Secure
// This single line prevents most CSRF attacks
ORIGIN / REFERER VALIDATION

The server can check the Origin or Referer header to verify the request came from the expected domain.

server-side validation logic
// Check Origin header on all state-changing requests
const origin = req.headers['origin'];
const allowed = ['https://bank.com'];
if (!allowed.includes(origin)) {
  return res.status(403).send('Forbidden');
}
// Origin cannot be spoofed by simple cross-site requests
Limitation: Referer can be stripped by privacy tools or HTTPS→HTTP redirects. Always use CSRF tokens as the primary defense, with Origin check as a secondary layer.
COMMON CSRF PROTECTION MISTAKES
MISTAKE 1
Token not validated server-side
Token exists in the form but the server never checks it. The check is missing or commented out.
MISTAKE 2
Same token for all users
Using a global static token instead of per-session random tokens. Once leaked, protects nobody.
MISTAKE 3
Token in URL parameter
Token in the URL leaks via Referer headers and browser history. Always use POST body or headers.
MISTAKE 4
Only protecting POST, not GET
State-changing GET requests are just as vulnerable. CSRF isn't just about POST forms.
MISTAKE 5
Token not tied to session
Token generated but not linked to user's session. Any valid token works for any user.
MISTAKE 6
Missing SameSite attribute
Not setting SameSite on cookies means browsers will send them on cross-site requests by default.
