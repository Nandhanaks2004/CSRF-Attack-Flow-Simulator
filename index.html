FOUNDATION MODULE

Cross-Site Request Forgery

CSRF tricks a logged-in user's browser into sending a request the user never intended. The server cannot tell the difference — because the browser makes it look legitimate.

🍪

The Cookie Problem

When you log into a website, the server gives you a session cookie. Your browser automatically sends that cookie with every subsequent request to that domain — whether you initiated the request or not.

This is by design. Browsers were built to trust session cookies. CSRF weaponises this trust.
⚠️

What CSRF Is

An attacker tricks your browser into sending a state-changing request to a site you're already logged into — without your knowledge or consent.

// You visit attacker's page
// Attacker's page contains:
<form action="https://bank.com/transfer">
  <input name="to" value="attacker"/>
  <input name="amount" value="5000"/>
</form>
// Auto-submits — your cookie is included!

Why Servers Can't Tell

The server sees a request from an authenticated user with a valid session cookie. From its perspective, this looks identical to a legitimate request made directly on its own site.

Core problem: HTTP was designed to be stateless. The session cookie is the only proof of identity — and the browser will send it automatically to any page that asks.
🌍

Real-World Impact

📧 Change victim's email address
🔑 Trigger password reset
💸 Make financial transactions

LAB 3 · PRESENCE-ONLY CHECK

CSRF — Token Validated Only If Present

The application has a CSRF token — but only validates it if it is submitted. If the token parameter is simply omitted, the server skips validation entirely and processes the request.

STEP 1
Capture the Normal Request
Normal POST
POST

email=test@test.com&csrf=ValidTokenABC
Looks like proper CSRF protection. Token is present and appears validated. But let's examine the server's logic…
STEP 2
The Logic Flaw (Server Side)

This is how the server actually validates the token — and it's critically flawed:

IF "csrf" parameter is present:
    IF token is valid:
        → allow request
    ELSE:
        → reject (403)
ELSE: // csrf param not in request at all
    → allow request ← 🚨 FLAW HERE
The fatal assumption: "If there's no token, there's nothing to validate." This backwards logic means an attacker simply removes the token field — triggering the no validation path.

DEFENSE MODULE

Preventing CSRF Attacks

Now that you've exploited four variants of CSRF, learn how to stop them. Effective defense requires multiple layers — no single control is sufficient on its own.

🎟

1. Proper CSRF Tokens

A CSRF token must be: random (high entropy), per-session (not reusable across sessions), validated on every state-changing request (all methods, all endpoints), and rejected if missing.

Server-Side Validation (Correct)
token = request.POST.get('csrf')
IF NOT token:
    raise 403 Forbidden  # missing = reject
IF token != session.csrf_token:
    raise 403 Forbidden  # wrong = reject
# Only now: process request
🍪

2. SameSite Cookie Attribute

Setting SameSite=Strict or SameSite=Lax on session cookies tells browsers not to send the cookie on cross-site requests.

Cookie Header
Set-Cookie: session=abc123;
  SameSite=Strict;
  HttpOnly; Secure; Path=/
ValueCross-Site Requests
StrictBlocked entirely
LaxGET top-level only
NoneAlways sent (insecure)
🔁

3. Re-Authentication for Critical Actions

For high-impact actions (password change, email change, fund transfer), require the user to re-enter their password. This cannot be forged by an attacker who only controls the victim's session.

Even if CSRF bypasses token checks, re-auth stops the attack — the attacker doesn't know the victim's password.
🌐

4. Origin / Referer Header Validation

Check that requests originate from your own domain. Cross-site form submissions will have a different Origin or Referer header.

Server Check
IF request.headers['Origin']
   != 'https://mysite.com':
    raise 403 Forbidden
Not foolproof — headers can sometimes be absent. Use as a secondary defense alongside tokens.
📮

5. Strict Method Enforcement

State-changing operations must only accept POST (or PUT/DELETE). Never allow GET to modify data. This prevents Lab 2-style bypasses.

Route Definition (Flask Example)
@app.route('/change-email',
    methods=['POST'])  # POST only
def change_email(): ...
📦

6. Double Submit Cookie Pattern

Set a random value as both a cookie and a form field. The server checks they match. Cross-site attackers can't read cookies (SOP), so they can't forge the form value to match.

Pattern
// Cookie: csrf-token=random123
// Form field: csrf=random123
// Server: if cookie === field → OK

SameSite=Strict is the strongest browser-side defense against CSRF.
📮

🧠

🛡
Defense in depth. Use CSRF tokens + SameSite + Origin check + re-auth for critical actions.
